# OpenClaw로 텔레그램 채널 자동운영: 수집→가공→초안 파이프라인 설계 실전

> 한 줄 요약: 자동화의 핵심은 “많이 가져오는 것”이 아니라, **노이즈를 줄이고 사람의 최종 의사결정 지점을 고정**하는 것이다.

---

## 1) 문제 정의: 왜 자동화해도 운영이 피곤해지는가

텔레그램/블로그 자동화를 붙이면 처음엔 편해지지만, 곧 두 가지 문제가 생긴다.

1. **노이즈 폭증**: 에러 로그/중복 시그널이 메시지 채널을 오염시킴  
2. **결정 지점 부재**: “무엇을 지금 쓸지”를 정해주는 단계가 없어 실행만 반복됨

실무에서 진짜 필요한 건 아래 3가지다.

- 신호 선별 기준(게이트)
- 실패 시 침묵 규칙(NO_REPLY)
- 하루 1회 사람 선택 포인트(저녁 토픽 픽)

---

## 2) 운영 아키텍처: 2-잡 구조로 분리

### A. Hot Signal Scout (백그라운드 정찰)
- 역할: AI/PM/C++/NAND 관련 신호 탐지
- 출력 원칙:
  - 엄격 게이트 통과 시만 요약 생성
  - 미충족/저신뢰 시 `NO_REPLY`
  - 툴 에러 로그는 사용자 출력 금지

### B. Evening Topic Pick Nudge (결정 트리거)
- 역할: 밤에 2~4개 후보를 제시하고 **한 개 선택** 받기
- 메시지 형식:
  - “오늘 드래프트로 쓸 핫시그널 1개 골라줘”
  - 번호 응답 유도
- 목적: 자동화의 마지막 10%를 사람 의사결정으로 고정

---

## 3) 게이트 설계: 소스 게이트 + 임팩트 게이트

핫시그널은 아래 둘 다 만족해야 한다.

### Source Gate
- 공식 소스 1개 이상
- 독립 메이저 미디어 2개 이상
- 접근 실패/페이월 소스는 검증 대상에서 제외

### Impact Gate
- AI/PM/C++/NAND 중 최소 1개 트랙에 **즉시 실무 영향**이 명확해야 함

이 구조를 쓰면 “뉴스 요약”이 아니라 “바로 글로 전환 가능한 신호”만 남는다.

---

## 4) 장애 패턴과 대응

### 패턴 1) 강제 실행 timeout
- 현상: `gateway timeout after 60000ms`
- 해석: 프런트 응답 timeout일 수 있으며, 백그라운드 실행은 완료될 수 있음
- 대응: 즉시 재실행보다 `cron runs` 후속 확인 우선

### 패턴 2) 검색/모델 rate limit
- 현상: Brave 429, provider cooldown
- 대응:
  - 요청 상한(run당 최대 요청 수) 설정
  - 소스 우선순위 고정(공식/접근 가능한 매체 우선)
  - 실패 로그는 사용자 채널 미노출

---

## 5) 실행 운영 규칙(권장)

1. **탐지와 알림을 분리**: 탐지는 조용히, 알림은 엄격 기준 통과 시만  
2. **하루 1회 의사결정**: 저녁에 하나만 고르게 설계  
3. **초안은 선택 이후 즉시 시작**: 선택→초안 사이 대기 제거  
4. **출력 포맷 고정**: 헤드라인 / 근거 / 링크 / 포스트 각도

## 7) 바로 적용 체크리스트

- [ ] Hot Signal job: 게이트 미충족 시 `NO_REPLY` 강제  
- [ ] 툴 에러 문자열 사용자 출력 금지 규칙 유지  
- [ ] Evening nudge: 후보 2~4개 + 번호 응답 유도  
- [ ] 선택 즉시 Draft job(또는 수동 초안) 트리거  
- [ ] 주간 KPI 기록으로 자동화 품질 추적

---

## 결론

자동화 파이프라인의 성패는 모델 성능보다 운영 설계에 달려 있다.  
핵심은 단순하다: **탐지는 자동화, 선택은 인간, 출력은 저노이즈**.

이 원칙만 지켜도 텔레그램 채널 운영은 “알림 폭탄”에서 “의사결정 도구”로 바뀐다.
